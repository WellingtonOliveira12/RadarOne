C) BACKEND — Conexão do Telegram via deep link + correção bot errado
1) Implementar “token de conexão”:
   - Endpoint autenticado: POST /api/telegram/connect-token
   - Retorna: { connectUrl, token, expiresAt }
   - connectUrl = https://t.me/RadarOneAlertaBot?start=connect_<token>
   - Token expira (ex.: 15 min ou 1h)
   - Salvar token no banco ligado ao userId e status “pending”
2) Bot handler:
   - Ao receber /start connect_<token>:
     - Validar token (existe, não expirado, status pending)
     - Vincular chatId + telegramUserId + username + firstName ao usuário
     - Marcar token como “used”
     - Responder no bot: “✅ Telegram conectado ao RadarOne com sucesso.”
     - (Opcional) enviar uma mensagem teste “Você receberá alertas aqui.”
3) Endpoint para status:
   - GET /api/telegram/status (auth)
   - Retorna: conectado? chatId? username? connectedAt?
4) Desconectar:
   - POST /api/telegram/disconnect (auth)
   - Limpa chatId/telegramUserId/username do usuário (ou marca como 
desconectado)
5) Cenário “Usuário conectou bot errado”:
   - Não dá para “corrigir” automaticamente do lado do RadarOne, porque o 
bot errado é externo.
   - Nossa correção é: permitir ao usuário gerar novo token e conectar no 
bot oficial.
   - UX: página/modal com passos:
     - “Abra o bot oficial pelo link”
     - “Clique em START”
     - “Aguarde a mensagem ✅ Conectado”
   - Backend deve permitir regenerar token e invalidar tokens anteriores.
6) Log/Auditoria:
   - Registrar eventos:
     - telegram_connect_token_created
     - telegram_connected
     - telegram_disconnected
     - telegram_connect_failed_invalid_token
     - telegram_connect_failed_expired
7) Segurança:
   - Token aleatório forte (ex.: 32+ chars)
   - Expiração
   - Um token só pode ser usado uma vez
   - Rate limit no endpoint de gerar token (evitar abuso)
8) Testes:
   - Unit tests (Vitest) para geração/validação token e fluxo de conexão
   - (Opcional) teste de integração simulado do handler
